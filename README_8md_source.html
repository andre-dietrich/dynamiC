16 ## Usage
17 
18 You can either generate a lib with `make lib` for dynamically linking or simply
19 include this repository into your project and include it into your Makefile.
20 
21 
22 ### Basic data types
23 
24 It is possible to define various basic data types as listed below:
25 
26 ```c
27 #include "dynamic.h"
28 
29 dyn_c var;
30 DYN_INIT(&var); // initialization with type NONE (undefined)
31 
32 dyn_set_bool(&var, DYN_TRUE);
33 dyn_set_int(&var, 3);
34 dyn_set_float(&var, 3.14159);
35 dyn_set_string(&var, "PI");
36 
37 dyn_set_extern(&var, void* anything ...);
38 ```
39 
40 
41 List have to be initialized with a default buffer length, this buffer is
42 automatically increased of decreased as the list grows or shrinks:
43 
44 ```c
45 dyn_c list;
46 DYN_INIT(&list);
47 
48 dyn_set_list_len(&list, 10);
49 dyn_length(&list); // results in 0
50 
51 dyn_list_push(&list, &var);
52 dyn_list_push(&list, &var);
53 dyn_length(&list); // results in 2
54 
55 // a placeholder to append elements of type NONE
56 dyn_list_push_none(&list);
57 dyn_list_push_none(&list);
58 dyn_length(&list); // results in 4
59 
60 // to refer to an element use
61 dyn_list_get(&list, 1);
62 dyn_list_get(&list, 2);
63 
64 // cleaning up
65 dyn_list_pop(&list);
66 dyn_list_pop(&list, 2);
67 ```
68 
69 Any element can be coppied with the following function, previously allocated
70 memory get cleaned up and new one gets created:
71 
72 ```c
73 dyn_copy(&list, &var); // dyn_copy(from, to);
74 dyn_type(&var) == LIST; // otherwise NONE, BOOL, INTEGER, FLOAT, STRING, ...
75 ```
76 
77 
78 Any type of dynamically allocated memory can be freed with one function, the
79 resulting elements are reset to type NONE:
80 
81 ```c
82 dyn_free(&list);
83 dyn_free(&var);
84 ```

86 ### Complex structures
87 
88 Further structures are sets and dictionaries, as for lists, memory is
89 automatically increased or decreased:
90 
91 ```c
92 dyn_c set;
93 DYN_INIT(&set);
94 dyn_set_set_len(&set, 1);
95 
96 DYN_INIT(&var);
97 dyn_set_insert(&set, &var);
98 dyn_set_insert(&set, &var);
99 dyn_set_insert(&set, &var);
100 
101 dyn_set_int(&var, -22);
102 dyn_set_insert(&set, &var);
103 dyn_set_insert(&set, &var);
104 dyn_set_insert(&set, &var);
105 
106 dyn_length(&set); // results in 2
107 dyn_get_string(&set); // "{,-22}" first element of set is of type NONE
108 ```
109 
110 
111 ```c
112 dyn_c dict;
113 DYN_INIT(&dict);
114 dyn_set_dict(&dict, 1);
115 
116 dyn_dict_insert(&dict, &var, "key");
117 dyn_dict_insert(&dict, &var, "another key");
118 dyn_dict_insert(&dict, &set, "key3");
119 
120 dyn_length(&dict); // results in 3
121 dyn_get_string(&dict); // "{'key': -22, 'another key': -22, 'key3': {,-22}}"
122 
123 var = dyn_dict_get(&dict, "key"); // get reference
124 
125 dyn_set_insert(&var, another dynamic element ...);
126 ```

128 ### Operations
129 
130 Operations are the commonly used `+, -, *, /, <, ==,... bitshift ...` among
131 others. All of them are defined within dynamic_op.h with the naming convention
132 `dyn_op_operator` with `(operator == add, sub, le, eq, not, pow, ...)`. Their
133 implementation mimics the common Python operators and the result of an operation
134 is stored within the first function parameter, which gets rewritten:
135 
136 ```c
137 dyn_c op1;
138 DYN_INIT(&op1);
139 dyn_set_int(&op1, 2);
140 
141 dyn_c op2;
142 DYN_INIT(&op1);
143 dyn_set_int(&op1, 2);
144 
145 dyn_op_add(&op1, &op2);
146 dyn_get_int(&op1); // returns 4
147 
148 dyn_op_pow(&op1, &op2);
149 dyn_get_int(&op1); // returns 16
150 ```
151 
152 Depending on the used datatypes, different behavior is exposed for some
153 operations, the highest datatype defines the result type, to which the other
154 parameter is casted, see the enumeration in dynamic_types.h.
155 
156 ```c
157 dyn_set_string(&op1, "string");
158 
159 dyn_op_add(&op1, &op2); // results in "string2"
160 ```
161 
162 Furhtermore, and unlike in Python, the order of parameters results in different
163 results:
164 
165 ```c
166 dyn_op_add(&op2, &op1); // results in "2string2"
167 
168 // or ...
169 dyn_op_add(&list, &element); // results in [ ..., element]
170 dyn_op_add(&element, &list); // results in [ element, ...]
171 ```
172 
173 Boolean operations can also be used for sets:
174 
175 ```c
176 dyn_op_and(&set1, &set2); // union
177 dyn_op_or(&set1, &set2); // difference
178 ```
179 
180 Otherwise, and as in Python, dynamic elements are cast to boolean values:
181 ```c
182 dyn_get_bool(&none); // DYN_FALSE
183 dyn_get_bool(&integer); // DYN_FALSE if zero, otherwise DYN_TRUE
184 dyn_get_bool(&float__); // DYN_FALSE if zero, otherwise DYN_TRUE
185 dyn_get_bool(&string); // DYN_FALSE if empty, otherwise DYN_TRUE
186 dyn_get_bool(&list); // DYN_FALSE if empty, otherwise DYN_TRUE
187 dyn_get_bool(&set); // DYN_FALSE if empty, otherwise DYN_TRUE
188 dyn_get_bool(&dict); // DYN_FALSE if empty, otherwise DYN_TRUE
189 
190 dyn_get_bool(&everythin_else); // DYN_FALSE
191 ```
192 
193 Boolean operations perform as usual, except if a value is NONE or associated
194 with... Then a ternary operations gets applied
195 
196 ```c
197 dyn_op_and(&true, &none); // NONE
198 dyn_op_and(&false, &none); // DYN_FALSE
199 
200 dyn_op_or(&true, &none); // DYN_TRUE
201 dyn_op_or(&false, &none); // NONE
202 
203 dyn_op_not(&none); // NONE
204 ```
205 
206 This behavior is quite usefull, if for example relational operations are
207 applied onto dynamic elements, which cannot be compared, such as:
208 
209 ```c
210 dyn_op_le(&string, &list); // NONE
211 dyn_op_gt(&procedure, &external); // NONE
212 ```
213 
214 The result element is of type NONE and can still be used in the further
215 evaluation ...
